# 环境记忆 — 捕获规则

**捕获规则**是"对这些会话,当触发条件满足时,触发 summarizer"。

## 按会话 vs 全局默认

`session_id` 控制范围:
- **NULL = 全局默认** — 应用于每个没有特定覆盖的活跃会话。
- **非 NULL = 按会话** — 仅对该会话生效,遮蔽全局默认。

大多数运维只有一个全局规则,从不设按会话的。覆盖存在是为这种情况: "本会话是手工慢审,更激进地总结它"。

## 触发类型

### `after_messages`(默认)

自上次捕获以来,有 N 条新用户消息到达后触发。最常见配置。

```
trigger_config: {"n": 6}
```

默认 n = 6。更小的数字(3)捕获更多事实但花更多 token;更大的(10-20)更便宜但更冒险(模型可能忘掉被中断的对话中段上下文)。

### `on_idle`

会话空闲(无新用户消息)至少 N 秒,且自上次触发以来存在新消息时触发。

```
trigger_config: {"seconds": 60}
```

对话成簇到来时有用 — 等簇落定后捕获整簇,而不是簇中。

### `k_chars`

新用户消息累计字符数跨过 K 时触发。

```
trigger_config: {"k": 4000}
```

默认 4000(约 1000 token)。对 prompt 长度差异很大的对话,这是 `after_messages` 的长度感知替代(长 prompt = 一次触发,十个短 prompt = 一次触发)。

### `manual`

从不自动触发。仅通过 **Run now** 按钮或 `POST /api/v1/memory-capture-rules/{id}/run-now` 触发。

对批处理 / 按需工作流有用,你不要持续捕获,但偶尔想把当前对话倒进记忆。

## 去重阈值

每条抽取出的事实在插入前会对目标作用域做 `memory.Search`。如果任何现有记忆返回相似度 > `dedup_threshold`(默认 0.85),该事实被跳过。

- 0.95 = 严格(只跳过近乎完全相同的表述)
- 0.85 = 推荐(可靠捕捉同义改述)
- 0.70 = 宽松(可能丢掉应该保留的事实;适合非常窄的项目)

被跳过/被存储的拆分记录在 `memory_summarizer_calls.facts_skipped_dedup` / `facts_stored`,这样你可以随时间调优。

## 目标作用域

抽取的事实存到哪:
- `session` — 仅本会话可见
- `project`(默认) — 同 cwd 下每个会话可见
- `global` — 所有地方可见

Project 对几乎每个工作流都合适。Global 对不绑特定项目的运维级偏好有用。

## 失败退避

当供应商对某 (rule, session) 组合连续失败 3 次,引擎会暂停该组合 1 小时。这防止配错的规则每个 tick 都烧 cycle。暂停在内存里 — 重启 opendray 会清掉。

## Run now

每个规则都有 **Run now** 按钮(UI)/ `POST /memory-capture-rules/{id}/run-now`(API),立即触发规则,绕过触发评估和暂停状态。对以下情况有用:
- 手动触发
- 测试刚创建的规则
- 结束会话前强制一次事实抽取
